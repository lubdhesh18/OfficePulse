# Contributing to OfficePulse

First of all, thank you for considering contributing to OfficePulse! 🎉

We appreciate every contribution, whether it's reporting bugs, suggesting improvements, improving documentation, or submitting code.


# Code of Conduct

Please read our Code of Conduct before contributing.

By participating in this project, you agree to follow its guidelines and maintain a respectful and welcoming environment.


# How Can I Contribute?

There are several ways to contribute to OfficePulse.

## Reporting Bugs

If you discover a bug:

- Search existing Issues before creating a new one.
- Describe the issue clearly.
- Include steps to reproduce.
- Mention your browser and operating system.
- Attach screenshots if applicable.


## Suggesting Features

Feature requests are always welcome.

When suggesting a feature:

- Explain the problem it solves.
- Describe the proposed solution.
- Keep suggestions aligned with the project's goals.


## Improving Documentation

Documentation improvements are greatly appreciated.

Examples include:

- README updates
- Grammar corrections
- Better explanations
- Additional diagrams
- Usage examples


## Contributing Code

Before writing code:

1. Fork the repository.

2. Clone your fork.

```bash
git clone https://github.com/lubdhesh18/OfficePulse.git
```

3. Create a new branch.

```bash
git checkout -b feature/feature-name
```

4. Make your changes.

5. Commit using meaningful commit messages.

Example:

```text
Add calendar attendance popup

Fix weekly summary calculation

Improve dashboard responsiveness
```

6. Push your branch.

```bash
git push origin feature/feature-name
```

7. Open a Pull Request.


# Coding Guidelines

Please follow these standards:

- Write clean and readable code.
- Keep functions modular.
- Use meaningful variable names.
- Avoid duplicate code.
- Comment only when necessary.
- Follow the existing project structure.

# Commit Message Convention

Use short and descriptive commit messages.

Examples:

```text
Add holiday API integration

Fix calendar selection bug

Improve UI responsiveness

Refactor storage module
```


# Pull Request Guidelines

Before submitting a Pull Request:

- Ensure the project builds successfully.
- Test your changes.
- Update documentation if required.
- Keep Pull Requests focused on one feature or fix.


# Development Philosophy

OfficePulse follows a documentation-first approach.

Major changes should align with the project's Software Requirements Specification (SRS) and Software Design Document (SDD).


# Questions?

If you have questions or need clarification, please open a GitHub Issue or Discussion.

Thank you for helping improve OfficePulse!
